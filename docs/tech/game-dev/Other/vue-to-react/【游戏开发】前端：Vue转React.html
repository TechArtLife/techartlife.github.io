【游戏开发】前端：Vue转React

缘由

最近，在开发图形系统时，逐渐意识到一件事的重要性：可视化调试信息。在一般的后端程序中，可以通过设置断点，逐步运行，添加监视变量，然后通过查看变量的值来检查程序的运行是否符合预期。但是，在涉及到图形系统开发的应用中，应用一般以30FPS的帧率运行，每一帧的数据都是重新生成的，应用出现的问题具有非常强的不可复现性。尤其是涉及到调试图形效果时，断点调试法已经无法满足图形系统开发需求。

所以，为了能够更加直观的对图形系统进行调试，提高图形系统的开发效率。我们将从最初开发应用时就添加可视化调试功能。最初的意思是指从零开始搭建图形系统，也就是说我们的可视化调试功能将覆盖我们从API初始化到绘制一个三角形的过程，并不是在构建完成图形系统之后进行的二次开发。也就是说，可视化调试功能和图形应用是完全独立的两个部分。

目前，我们了解到的解决类似问题的项目中，有一个比较符合我们的要求： Tracy Profiler。基本原理是：目标应用程序通过调用Tracy的API将一帧中需要统计的数据通过网络发送到Tracy Profiler这个软件中（充当一个服务器）。Tracy Profiler的界面使用Dear IMGUI实现，性能应该不错。这个库也在一些工业级的项目中有用到，例如Adobe家的某产品。

我们也尝试了一下这个项目，体验就是：虽然官方文档写的很详细，但是入门还是需要一段时间好好研究才行，这对我们来说又是一个不可控的外部变量（哭）。另一方面，这个库比较适合已经完成初始图形框架搭建的项目使用，但是我们的项目目前还处于API组合阶段，所以也就没有任何图形框架可用。最终，我们决定用我们自己的技能树来完成这个功能。反正项目刚诞生，用任何已有技能树实现需求都是可以的。

同时，贯彻我们的执行准则：

1.反复确认首要目标和次要目标。 2.不熟悉需求与具体逻辑实现的时候，绝不做大规模架构设计。 3.在学习过程中，早期的架构设计一定会被推翻，而且肯定会限制后期的功能开发。

以此来推出我们现在要做的事情就是：用我们的技能树（树叶）来开发可视化调试功能。我们暂且称之为功能，后期功能成熟了，我们再给他改名，嘿嘿。

实现

那么，现在就该考虑使用技能树上的哪些技能点来实现它了。很明显，通过网络将可视化功能和应用隔离开是目前最好的方案。应用和可视化功能可以运行在同一台设备上，也可以运行在两台设备上。

那么我们现在的技能树是什么情况呢：

  • Vue前端框架：使用Electron+Vue做过两个项目，有前端HTML/CSS/JavaScript使用经验。

看起来，技能树的生长潜力十分巨大呢:P

使用了这么点时间的Vue后，我们觉得基于模板的前端框架不太适合像我们这样以C/C++为主的开发者。所以我们的目光转向了另一个非常火的前端框架 React。

经过短暂的了解，我们获取的信息大致如下：

  • Vue: 基于模板；适合新手和传统HTML开发者；有一些功能封装；
  • React: 基于函数；可能适合像我们这样偏后端的开发者；可以获得更高的控制权；

所以，我们决定给前端技能树一个生长的机会，从Vue转向React。

挑战

  • React：一段新的征程。
  • 网络库： 计划使用跨平台的libevent，在其基础上封装一些功能。

总结

可视化征途出发！
